Berikut penjelasan lengkap isi file flowchart E0 Baseline Experimental Protocol beserta tahapan penelitiannya secara sistematis.

1. Gambaran Umum Penelitian
File ini mendeskripsikan protokol eksperimen baseline (E0) untuk:


Task: Object Detection + Classification (YOLO)


Domain: Tandan buah sawit (FFB)


Input: RGB-only


Output: 4 kelas kematangan (B1–B4 / M1–M4)


Durasi: ~4 minggu
Tujuan utama:


Menentukan baseline kuat


Mengidentifikasi bottleneck utama (data vs model vs pipeline)


Menghindari trial random tanpa arah

2. Struktur Tahapan Penelitian (High-Level)
Penelitian dibagi menjadi 4 fase utama + decision framework:
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Decision

3. PHASE 0 — Validation & Calibration (±3 hari)
Tujuan: memastikan data valid sebelum training serius
Task A — Exploratory Data Analysis (EDA)
Isi:


Distribusi kelas (B1–B4)


Ukuran objek (terutama B4 kecil)


Cek kualitas label (20 gambar terburuk)


Split data berbasis pohon (tree-grouped)
Insight penting:


Jika objek terlalu kecil (<16px), resolusi rendah gagal


Split harus by tree, bukan per gambar (hindari data leakage)

Task B — Resolution Sweep
Eksperimen:


Bandingkan:


640px vs 1024px


2 seeds
Keputusan:



5% improvement → pakai 1024



2–5% → pertimbangan device


<2% → tetap 640

Task C — Data Learning Curve
Eksperimen:


Train dengan:


25% / 50% / 75% / 100% data
Tujuan:


Menentukan apakah:


Model sudah “saturasi”


Atau masih butuh data

Gate (KRITIS)
Jika >10% label error → STOP (relabel dataset)

Output Phase 0


Split dataset final (70/15/15)


Resolusi fix


Status kecukupan data


Validasi kualitas label

4. PHASE 1A — Pipeline Decision (±2–3 hari)
Tujuan: menentukan arsitektur pipeline
Dua opsi:
1. One-stage


YOLO langsung klasifikasi 4 kelas
2. Two-stage


Step 1: deteksi FFB (1 kelas)


Step 2: klasifikasi kematangan

Evaluasi:


mAP


Confusion B2/B3 (kelas mirip)

Keputusan:



5% improvement → two-stage



<2% → one-stage


2–5% → judgement

Output:


Pipeline final


Jika 2-stage → classifier terbaik

5. PHASE 1B — Architecture Sweep (±1.5 minggu)
Tujuan: mencari model terbaik
Eksperimen:


11 model:


YOLOv8 (n/s/m)


YOLOv9


YOLOv10


YOLO26


YOLOv11




2 seeds tiap model

Parameter tetap:


lr = 0.001


batch = 16


augmentasi medium

Fokus analisis:


Kenapa model terbaik menang?


Semua model gagal di mana?

Gate:
Jika mAP < 70% → STOP (data bermasalah)

Output:


Top 2–3 model


Analisis kegagalan


Feasibility kelas sulit (B4)

6. PHASE 2 — Hyperparameter Optimization (±1 minggu)
Tujuan: optimasi parameter training
Pendekatan:


Sequential tuning (bukan random)

Urutan tuning:
Step 0a — Imbalance


No weighting


Class weighting


Focal loss
Step 0b — Ordinal (penting!)


Standard CE


Ordinal loss


CORAL (jika 2-stage)
Step 1 — Learning Rate


0.0005 / 0.001 / 0.002
Step 2 — Batch size


8 / 16 / 32
Step 3 — Augmentation


Light / Medium / Heavy

Prinsip:


Lock parameter satu per satu


Total ~80 eksperimen

Decision:
Jika improvement <1% → kembali ke baseline Phase 1

Output:


Konfigurasi terbaik


Loss + LR + batch + augmentasi optimal

7. PHASE 3 — Final Validation (±3 hari)
Tujuan: finalisasi model sebelum deploy

Step 1 — Retrain


Train ulang:


Train + Val digabung


Epoch: 100–150


No early stopping

Step 2 — Deploy Check


Export ke:


TFLite




Constraint:


<100MB


cepat di tablet

Step 3 — Threshold Optimization


Sweep confidence:


0.1 – 0.5
Optimasi:


mAP


confusion B2/B3


recall B4

Step 4 — Error Analysis
Analisis 20 error terburuk:


objek kecil miss


confusion antar kelas


occlusion


false positive


label error

Gate lagi:
Jika >10% error = label salah → STOP (relabel)

Output:


Final metric


Confusion matrix


Error breakdown


Curve threshold

8. Decision Framework (Final)
Model dinilai dengan dual-gate:


mAP@0.5


confusion B2/B3


semua kelas ≥70%

9. Inti Filosofi Protokol Ini
Ini bukan sekadar training YOLO biasa, tapi:
1. Data-first approach


Validasi data sebelum model
2. Controlled experiment


Semua variabel dikunci step-by-step
3. Decision-based research


Ada gate STOP jelas


Tidak buang waktu compute
4. Fokus root cause


Apakah masalah:


data?


resolusi?


arsitektur?


hyperparameter?

10. Kesimpulan Singkat
Protokol ini adalah:


Pipeline riset terstruktur


Menghindari trial-error acak


Memastikan:


baseline kuat


insight ilmiah jelas


keputusan berbasis data
