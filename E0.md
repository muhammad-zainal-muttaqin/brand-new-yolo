# E0 Baseline Experimental Protocol

Dokumen ini adalah versi Markdown dari protokol baseline E0. Sumber canonical-nya ada pada flowchart berikut:

- `https://github.com/muhammad-zainal-muttaqin/YOLOBench/blob/main/E0_Protocol_Flowchart.html`

Untuk aturan operasional khusus repo ini, buka [GUIDE.md](GUIDE.md). Untuk konteks keputusan riset, buka [CONTEXT.md](CONTEXT.md).

> Catatan nomenklatur repo ini: flowchart memakai `M1–M4`, sedangkan dataset aktif di workspace ini memakai `B1–B4`. Untuk repo ini, mapping yang sah adalah: `B1 = paling matang`, lalu turun sampai `B4 = paling belum matang`.
>
> Panduan visual yang dipakai konsisten di repo ini:
> - `B1`: buah **merah**, **besar**, **bulat**, posisi **paling bawah** pada tandan
> - `B2`: buah masih **hitam**, mulai **transisi ke merah**, sudah **besar** dan **bulat**, posisi **di atas B1**
> - `B3`: buah **full hitam**, masih **berduri**, masih **lonjong**, posisi **di atas B2**
> - `B4`: buah **paling kecil**, **paling dalam di tandan**, sulit terlihat, masih banyak **duri**, warna **hitam sampai hijau**

## 1. Gambaran umum

- **Task**: RGB-only object detection untuk 4 kelas kematangan
- **Domain**: oil palm bunch / FFB
- **Output**: 4 kelas kematangan (`M1–M4`, di repo ini dipetakan ke `B1–B4`)
- **Estimasi durasi**: ~4 minggu
- **Ritme keputusan**: gate review di akhir tiap fase

Tujuan E0:

- membangun baseline yang kuat dan terkontrol
- mengidentifikasi bottleneck utama sejak awal
- mencegah eksperimen acak yang sulit dibandingkan

## 2. Struktur besar protokol

Urutan eksperimen canonical:

1. **Phase 0 — Validation & Calibration**
2. **Phase 1A — Pipeline Decision**
3. **Phase 1B — Architecture Sweep**
4. **Phase 2 — Hyperparameter Optimization**
5. **Phase 3 — Final Validation**
6. **Decision Framework**

## 3. Phase 0 — Validation & Calibration (~3 hari)

Tujuan fase ini sederhana: memastikan data valid, split sah, resolusi kerja masuk akal, dan kecukupan data mulai terbaca sebelum training serius.

### Task A — Exploratory Data Analysis

Lakukan minimal:

- distribusi kelas
- analisis ukuran objek
- perhatian khusus pada objek minimum `M4`
- label quality spot-check pada **20 gambar** yang paling problematik

### Task B — Resolution Sweep

Eksperimen canonical:

- model: `YOLOv8s`
- resolusi: `640` vs `1024`
- seeds: `2`

Aturan lock resolusi:

- jika gain `>5% mAP`, lock `1024`
- jika gain kecil atau biaya device tinggi, pilih `640`

### Task C — Data Learning Curve

Eksperimen canonical:

- model: `YOLOv8s`
- subset training: `25% / 50% / 75% / 100%`
- stratified by class
- seeds: `2`

Analisis yang wajib:

- plot learning curve
- per-class AP (`M1/M2/M3/M4` terpisah)
- cek apakah `M4` masih terus naik saat data bertambah

### Output / lock akhir Phase 0

Pada akhir Phase 0, yang dikunci adalah:

- split final: tree-grouped, class-stratified (`70/15/15`)
- resolusi final (`640` atau `1024`)
- data sufficiency assessment per kelas
- baseline class weighting strategy
- hasil label quality check

### Gate Phase 0

- jika **>10% label errors** → **STOP** dan relabel dulu
- jika dataset cukup bersih → lanjut ke Phase 1A

## 4. Phase 1A — Pipeline Decision (~2–3 hari)

Tujuan fase ini adalah memilih pipeline yang akan dipakai sampai akhir: **one-stage** atau **two-stage**.

### Opsi yang dibandingkan

#### One-stage

YOLO langsung melakukan 4-class detection + classification.

#### Two-stage

- Stage 1: binary FFB detector
- Stage 2: crop → classifier
- kandidat classifier di flowchart:
  - `EfficientNet-B0`
  - `ResNet-18`
  - `YOLO-cls`

### Setup eksperimen canonical

- `2–3` representative architectures
- uji **dua pipeline**
- `2 seeds`

### Kriteria keputusan

Bandingkan pipeline memakai dual-gate berikut:

- mAP
- confusion `M2/M3`
- confusion `M3/M4`

Catatan penting: pada related dataset, two-stage pernah mengungguli one-stage untuk maturity classification. Karena itu, two-stage perlu **diuji**, bukan langsung diasumsikan kalah.

### Aturan keputusan akhir Phase 1A

- jika confusion improvement `>5%` → pilih **two-stage**
- jika `<2%` → pilih **one-stage**
- jika `2–5%` → pakai judgement call
- jika hasil masih noisy → tambah seeds

### Output / lock akhir Phase 1A

- pipeline final (`one-stage` atau `two-stage`)
- jika yang menang `two-stage`, lock juga **best Stage-2 classifier**

## 5. Phase 1B — Architecture Sweep (~1.5 minggu)

Tujuan fase ini adalah mencari arsitektur terbaik **di dalam pipeline pemenang** dari Phase 1A.

### Prinsip canonical

- jika pipeline pemenang adalah **one-stage**, sweep dilakukan sebagai **4-class detector**
- jika pipeline pemenang adalah **two-stage**, sweep dilakukan sebagai **binary detector + classifier variants**

### Roster arsitektur canonical

Total **11 architectures × 2 seeds**:

- **YOLOv8**: `n / s / m`
- **YOLOv9**: `c`
- **YOLOv10**: `n / s / m`
- **YOLO26**: `n / s / m`
- **YOLOv11**: `m`

### Hyperparameter default selama sweep

- `lr0 = 0.001`
- `batch = 16`
- `medium augmentation`
- `patience = 15–25`
- tracking wajib per kelas `M1/M2/M3/M4`

### Analisis wajib di akhir Phase 1B

- mengapa winner menang
- di mana semua model sama-sama gagal
- error stratification pada **20 gambar terburuk** untuk **top-3 architectures**
- perhatian khusus pada confusion `M2/M3` dan `M3/M4`
- feasibility assessment untuk `M4`

### Output Phase 1B

- top `2–3` architectures di pipeline pemenang
- `M4` feasibility assessment
- failure mode categorization

### Gate Phase 1B

- jika **best mAP < 70%** → **STOP**
- jika `>= 70%` → lanjut ke Phase 2

## 6. Phase 2 — Hyperparameter Optimization (~1 minggu)

Tujuan fase ini adalah tuning terstruktur pada **top 2–3 architectures** dari Phase 1B.

### Prinsip canonical

- pakai **sequential search**, bukan random search
- tuning dilakukan pada **top 2–3 architectures × 2 seeds**
- **satu parameter harus dikunci sebelum pindah ke parameter berikutnya**

### Urutan tuning canonical

#### Step 0a — Imbalance
- no weighting
- class-weighted
- focal (`γ = 1.5`)

#### Step 0b — Ordinal
- standard CE
- ordinal-weighted
- CORAL

> `CORAL` hanya relevan atau tersedia untuk pipeline **two-stage**.

#### Step 1 — Learning Rate
- `0.0005`
- `0.001`
- `0.002`

#### Step 2 — Batch Size
- `8`
- `16`
- `32`

#### Step 3 — Augmentation
- `Light`
- `Medium`
- `Heavy`

### Analisis akhir Phase 2

Yang harus dijawab:

- apakah tuning benar-benar memperbaiki baseline Phase 1
- apakah imbalance handling atau ordinal handling membantu confusion `M2/M3` dan `M3/M4`
- jika average improvement `<1%`, apakah lebih aman kembali ke baseline terbaik Phase 1

### Output / lock akhir Phase 2

- best config per architecture
- kombinasi final untuk loss, LR, batch, dan augmentation
- finalists dengan `3+ seeds` untuk confidence yang lebih baik

## 7. Phase 3 — Final Validation (~3 hari)

Tujuan fase ini adalah finalisasi model sebelum keputusan deploy.

### Step 1 — Retrain

- train memakai **Train + Val Combined**
- pakai hyperparameter yang sudah dikunci dari Phase 2
- canonical reference: `epochs 100–150`
- override operasional repo ini untuk sesi sekarang: `epochs=60`, `patience=15`, `min_epochs=60`

Dengan override ini, run final tetap berjalan penuh sampai minimal `60` epoch.

### Step 2 — Deploy Check

Flowchart canonical tetap mencantumkan deploy check. Namun pada repo ini, override operasionalnya adalah:

- amankan `best.pt` dulu
- export ke **TFLite** boleh ditunda
- **optional INT8 quantization** juga boleh ditunda
- tablet test boleh dilakukan nanti saat hardware siap

Jika konversi deploy dilakukan belakangan, akurasi, ukuran, latency, dan kompatibilitas device **wajib diuji ulang** pada artefak hasil konversi.

### Step 3 — Calibrate

- lakukan confidence threshold sweep pada rentang `[0.1 – 0.5]`
- optimasi berdasarkan dual-gate:
  - `mAP@0.5`
  - confusion `M2/M3`
- track juga **`M4` recall**
- langkah ini **tidak boleh** mengubah training

### Step 4 — Analyze

Lakukan **error stratification** pada **20 gambar terburuk** dan kelompokkan minimal ke:

- small object missed
- confusion `M2/M3`
- occlusion
- false positive
- lighting
- label error

### Output Phase 3

- final metrics pada threshold terbaik
- confusion matrix 4 kelas
- error stratification table
- threshold-vs-metric curves

Hasil error analysis ini perlu dipetakan ke root cause dan dipakai untuk memprioritaskan eksperimen lanjutan jika baseline masih kurang.

## 8. Decision framework final

Keputusan akhir memakai dual-gate:

- `mAP@0.5`
- confusion `M2/M3`
- semua kelas `>= 70%`

| Level | Kondisi | Aksi |
|---|---|---|
| **EXCELLENT** | `mAP ≥ 90%` · `M2/M3 < 20%` · semua kelas `≥70%` | Deploy. Skip eksperimen lanjutan. |
| **GOOD** | `mAP 85–90%` · `M2/M3 20–30%` · semua kelas `≥70%` | Deploy. Eksperimen lanjutan opsional. |
| **ACCEPTABLE** | `mAP 80–85%` · `M2/M3 > 30%` · semua kelas `≥70%` | Deploy baseline. Eksperimen lanjutan wajib. |
| **NEEDS WORK** | `mAP 75–80%` **atau** ada kelas `<70%` | Jangan deploy. Jalankan eksperimen lanjutan. |
| **INSUFFICIENT** | `mAP < 75%` | STOP. Audit data quality, class definitions, dan feasibility task. |

## 9. Filosofi singkat protokol ini

1. **Data-first** — validasi data sebelum membahas model
2. **Controlled experiment** — ubah satu hal, lalu kunci
3. **Decision-based research** — tiap fase harus menghasilkan keputusan
4. **Root-cause oriented** — kegagalan harus dipetakan ke penyebabnya

## 10. Ringkasan singkat

E0 adalah baseline protocol yang:

- terstruktur
- berorientasi keputusan
- menjaga eksperimen tetap apple-to-apple
- menghasilkan baseline yang bisa dipakai sebelum membuka eksperimen lanjutan

Untuk repo ini, [E0.md](E0.md) adalah **protokol canonical**, sedangkan [GUIDE.md](GUIDE.md) adalah **runbook operasional** yang menjelaskan policy lokal, override, dan implementasi nyata di workspace ini.
