# E0 Baseline Experimental Protocol

Sumber canonical protokol ini adalah flowchart:
- `https://github.com/muhammad-zainal-muttaqin/YOLOBench/blob/main/E0_Protocol_Flowchart.html`

Dokumen ini adalah versi Markdown yang disinkronkan dengan flowchart tersebut agar bisa dipakai sebagai referensi tekstual di repo ini.

> Catatan nomenklatur repo ini: flowchart memakai istilah kelas `M1–M4`, sedangkan dataset aktif di workspace ini memakai `B1–B4`. **Source of truth operasional untuk repo ini adalah label dataset `B1–B4`, dengan arti biologis yang tidak boleh tertukar:** `B1 = buah paling matang / ripe`, lalu berurutan turun tingkat kematangannya sampai `B4 = buah paling mengkal / belum matang`. Jadi urutan biologis `B1 → B4` bergerak dari **paling matang** ke **paling belum matang**.
>
> Panduan domain visual/posisional yang harus dipakai konsisten di repo ini:
> - `B1`: buah **merah**, **besar**, **bulat**, dan posisinya **paling bawah** pada tandan; ini menandakan kematangan tertinggi.
> - `B2`: buah masih **hitam** namun sudah mulai **transisi ke merah**; buah juga sudah **besar** dan **bulat**; posisinya **di atas B1**.
> - `B3`: buah **full hitam**, masih ada **duri** pada buah, bentuknya masih **lonjong**, dan posisinya **di atas B2**.
> - `B4`: buah **paling kecil**, **paling dalam di batang/tandan**, sering **susah terlihat**, masih banyak **duri**, warnanya **hitam bahkan hijau**, dan buah pada tandannya masih bisa berkembang lebih besar; ini kelas **paling belum matang**.

---

## 1. Gambaran Umum

- **Task**: RGB-only object detection untuk 4 kelas kematangan
- **Domain**: oil palm bunch / FFB
- **Output**: 4 kelas kematangan (`M1–M4`, pada repo ini dipetakan ke `B1–B4`)
- **Estimasi durasi**: ~4 minggu
- **Ritme keputusan**: end-of-phase meeting / gate review pada setiap akhir fase

Tujuan E0:
- membangun baseline kuat dan terkontrol,
- mengidentifikasi bottleneck utama (data, resolusi, pipeline, arsitektur, atau hyperparameter),
- mencegah trial-and-error yang acak.

---

## 2. Struktur Besar Protokol

Urutan eksperimen canonical:

1. **Phase 0 — Validation & Calibration**
2. **Phase 1A — Pipeline Decision**
3. **Phase 1B — Architecture Sweep**
4. **Phase 2 — Hyperparameter Optimization**
5. **Phase 3 — Final Validation**
6. **Decision Framework**

---

## 3. Phase 0 — Validation & Calibration (~3 hari)

Tujuan: memastikan data valid, split sah, resolusi tepat, dan kecukupan data dipahami sebelum training serius.

### Task A — Exploratory Data Analysis
Lakukan:
- distribusi kelas,
- analisis ukuran objek,
- perhatian khusus pada dimensi minimum objek `M4`,
- label quality spot-check pada **20 gambar teratas** yang paling problematik.

### Task B — Resolution Sweep
Eksperimen canonical:
- `YOLOv8s`
- `640` vs `1024`
- `2 seeds`

Aturan lock resolusi:
- **gain >5% mAP** → lock `1024`
- gain kecil / device cost tinggi → pertimbangkan `640`

### Task C — Data Learning Curve
Eksperimen canonical:
- `YOLOv8s`
- subset training `25% / 50% / 75% / 100%`
- **stratified by class**
- `2 seeds`

Analisis yang wajib:
- plot learning curve,
- per-class AP (`M1/M2/M3/M4` terpisah),
- cek apakah `M4` masih terus naik; bila iya, arahkan ke targeted data collection.

### Output / lock di akhir Phase 0
Pada akhir Phase 0, yang dikunci adalah:
- **split** final: tree-grouped, class-stratified (`70/15/15`),
- **resolution** final (`640` atau `1024`),
- **data sufficiency assessment** per kelas,
- **class weighting strategy** baseline,
- **label quality check**.

### Gate Phase 0
- Jika **>10% label errors** → **STOP** dan lakukan relabel dataset sebelum lanjut.
- Jika dataset bersih → lanjut ke Phase 1A.

---

## 4. Phase 1A — Pipeline Decision (~2–3 hari)

Tujuan: memutuskan apakah pipeline terbaik adalah **one-stage** atau **two-stage**.

### Opsi yang dibandingkan
#### One-stage
- YOLO langsung melakukan 4-class detection + classification.

#### Two-stage
- Stage 1: binary FFB detector
- Stage 2: crop → classifier
- kandidat classifier yang dicantumkan flowchart:
  - `EfficientNet-B0`
  - `ResNet-18`
  - `YOLO-cls`

### Setup eksperimen canonical
- `2–3` representative architectures
- uji **kedua pipeline**
- `2 seeds`

### Kriteria keputusan
Bandingkan pada **dual-gate**:
- mAP,
- confusion `M2/M3`,
- confusion `M3/M4`.

Catatan flowchart:
- pada related dataset, two-stage pernah mengungguli one-stage untuk maturity classification; karena itu two-stage harus benar-benar diuji, bukan diasumsikan kalah.

### Aturan keputusan akhir Phase 1A
- **>5% confusion improvement** → pilih **two-stage**
- **<2%** → pilih **one-stage**
- **2–5%** → judgement call
- jika hasil masih berada dalam noise → tambah seeds

### Output / lock di akhir Phase 1A
- **pipeline final** (`one-stage` atau `two-stage`)
- jika `two-stage`, lock juga **best Stage-2 classifier**

---

## 5. Phase 1B — Architecture Sweep (~1.5 minggu)

Tujuan: mencari arsitektur terbaik **di dalam pipeline pemenang** dari Phase 1A.

### Prinsip canonical
- Jika pipeline pemenang adalah **one-stage**, lakukan sweep sebagai **4-class detectors**.
- Jika pipeline pemenang adalah **two-stage**, lakukan sweep sebagai **binary detectors** + classifier variants.

### Roster arsitektur canonical
Total **11 architectures × 2 seeds**:
- **YOLOv8**: `n / s / m`
- **YOLOv9**: `c`
- **YOLOv10**: `n / s / m`
- **YOLO26**: `n / s / m`
- **YOLOv11**: `m`

### Hyperparameter default yang harus tetap selama sweep
- `lr0 = 0.001`
- `batch = 16`
- `medium augmentation`
- `patience = 15–25`
- tracking wajib per kelas `M1/M2/M3/M4`

### Analisis wajib akhir Phase 1B
- mengapa winner menang,
- di mana **semua model** gagal,
- error stratification pada **20 gambar terburuk** untuk **top-3 architectures**,
- perhatian khusus pada confusion `M2/M3` dan `M3/M4`,
- feasibility assessment untuk `M4`.

### Output Phase 1B
- **Top 2–3 architectures** di dalam pipeline pemenang,
- **M4 feasibility assessment**,
- **failure mode categorization**.

### Gate Phase 1B
- Jika **best mAP < 70%** → **STOP**
- Jika `>=70%` → lanjut ke Phase 2

---

## 6. Phase 2 — Hyperparameter Optimization (~1 minggu)

Tujuan: tuning terstruktur pada **top 2–3 architectures** dari Phase 1B.

### Prinsip canonical
- **Sequential search**, bukan random search.
- Tuning dilakukan pada **top 2–3 architectures × 2 seeds**.
- **Setiap parameter harus di-lock sebelum pindah ke parameter berikutnya.**

### Urutan tuning canonical
#### Step 0a — Imbalance
- No weighting
- Class-weighted
- Focal (`γ = 1.5`)

#### Step 0b — Ordinal
- Standard CE
- Ordinal-weighted
- CORAL

> `CORAL` hanya tersedia / relevan untuk pipeline **two-stage**.

#### Step 1 — Learning Rate
- `0.0005`
- `0.001`
- `0.002`

#### Step 2 — Batch Size
- `8`
- `16`
- `32`

#### Step 3 — Augmentation
- `Light`
- `Medium`
- `Heavy`

### Analisis akhir Phase 2
- apakah optimasi benar-benar memberi pengaruh dibanding baseline Phase 1,
- apakah imbalance handling atau ordinal handling menjadi lever yang lebih besar untuk `M2/M3` dan `M3/M4`,
- jika average improvement `<1%`, kembali ke konfigurasi terbaik dari Phase 1.

### Output / lock di akhir Phase 2
- **best config per architecture**,
- kombinasi final untuk:
  - loss (imbalance + ordinal),
  - learning rate,
  - batch size,
  - augmentation,
- **finalists dengan 3+ seeds** untuk statistical confidence.

---

## 7. Phase 3 — Final Validation (~3 hari)

Tujuan: finalisasi model sebelum keputusan deploy.

### Step 1 — Retrain
- Train menggunakan **Train + Val Combined**
- pakai semua data berlabel
- hyperparameter dibekukan dari hasil Phase 2
- `epochs 100–150`
- **no early stopping**

### Step 2 — Deploy Check
Canonical flowchart tetap mencantumkan deploy check ini. Namun untuk override operasional repo ini pada sesi sekarang:
- final `best.pt` diprioritaskan untuk diamankan lebih dulu,
- export ke **TFLite** boleh ditunda,
- **optional INT8 quantization** juga boleh ditunda,
- tablet test boleh dilakukan nanti saat hardware / kebutuhan deploy sudah siap.

Catatan penting: bila konversi deploy dilakukan belakangan, artefak hasil konversi itu tetap harus menjalani validasi ulang akurasi, ukuran, latency, dan kompatibilitas device.

### Step 3 — Calibrate
- confidence threshold sweep pada rentang `[0.1 – 0.5]`
- optimasi berdasarkan dual-gate:
  - `mAP@0.5`
  - confusion `M2/M3`
- track juga **M4 recall**
- langkah ini **tidak boleh retraining**

### Step 4 — Analyze
Lakukan **error stratification** pada **20 gambar terburuk** dan kategorikan minimal ke:
- small object missed,
- `M2/M3` confusion,
- occlusion,
- false positive,
- lighting,
- label error.

### Output Phase 3
- final metrics pada threshold terbaik,
- 4-class confusion matrix,
- error stratification table,
- threshold-vs-metric curves.

### Penutup akhir fase
Hasil error analysis harus dipetakan ke root cause dan dipakai untuk memprioritaskan eksperimen lanjutan (`E1–E6`) bila masih dibutuhkan.

---

## 8. Decision Framework (Final)

Keputusan akhir memakai **dual-gate**:
- `mAP@0.5`
- confusion `M2/M3`
- semua kelas `>=70%`

### Kategori hasil
| Level | Kondisi | Aksi |
|---|---|---|
| **EXCELLENT** | `mAP ≥ 90%` · `M2/M3 < 20%` · semua kelas `≥70%` | Deploy. Skip `E1–E6`. |
| **GOOD** | `mAP 85–90%` · `M2/M3 20–30%` · semua kelas `≥70%` | Deploy. `E1–E3` opsional. |
| **ACCEPTABLE** | `mAP 80–85%` · `M2/M3 > 30%` · semua kelas `≥70%` | Deploy baseline. `E1–E3` wajib. |
| **NEEDS WORK** | `mAP 75–80%` **atau** ada kelas `<70%` | Jangan deploy. Jalankan penuh `E1–E6`. |
| **INSUFFICIENT** | `mAP < 75%` | STOP. Investigasi data quality, class definitions, dan task feasibility. |

---

## 9. Inti Filosofi Protokol

1. **Data-first**
   - validasi data sebelum model.
2. **Controlled experiment**
   - variabel dikunci step-by-step.
3. **Decision-based research**
   - ada gate jelas di akhir tiap fase.
4. **Root-cause oriented**
   - kegagalan harus dipetakan ke data, resolusi, arsitektur, loss, atau optimasi.

---

## 10. Ringkasan Singkat

E0 adalah baseline protocol yang:
- terstruktur,
- berorientasi keputusan,
- menjaga eksperimen tetap apple-to-apple,
- menghasilkan baseline kuat sebelum masuk eksperimen lanjutan.

Untuk repo ini, `E0.md` adalah **referensi protocol canonical**, sedangkan `GUIDE.md` adalah **runbook operasional** yang boleh memuat catatan implementasi lokal, policy Git, policy artefak, dan override eksekusi yang sudah diputuskan secara eksplisit selama sesi kerja.
